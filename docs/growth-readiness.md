# LOCAL AI OS growth readiness

Status: incubation.

The public site is a product and documentation shell. It is not yet evidence
that an external customer will buy, activate, retain, or refer the product.

## Decision

Do not buy traffic for the current product shell.

Before acquisition, select one narrow external workflow for one buyer. The
workflow must have a measurable baseline, a bounded implementation, and an
acceptance result that can be shown without exposing private machine data.

## First commercial shape

```text
one technical owner
-> one costly repeated context or recovery problem
-> 9,900 RUB diagnostic
-> measured before/after proof
-> 49,900 RUB bounded pilot
-> acceptance report
-> support or referral
```

The first wedge is not "install an AI operating system". Candidate wedges must
be phrased as an observable job, for example:

- recover project context and the next action after a restart;
- retrieve a verified answer from one private knowledge source with permissions;
- route one repeated engineering task to a measured local model and produce an
  acceptance report.

Choose only one after external interviews.

## Promotion gate

- [ ] one external ICP and decision maker;
- [ ] one repeated expensive workflow;
- [ ] baseline and acceptance metric agreed before work;
- [ ] one sanitized client proof bundle;
- [ ] delivery and recovery boundaries verified;
- [ ] attributed qualified-request path;
- [ ] public seller/support identity;
- [ ] explicit payment and receipt path.

Until these gates pass, use founder education and problem interviews, not paid
traffic or broad autonomy claims.

### Founder-led paid diagnostic cohort

The incubation boundary does not forbid selling the bounded diagnostic that
creates the first external evidence. The first cohort is limited to three
technical owners. Each request must arrive with the Telegram intake marker
`LOCAL AI OS / diagnostic-2026-07`, name one repeated workflow and agree a
baseline plus acceptance metric before payment. Do not present the cohort as a
validated general-purpose AI OS or expand it into paid acquisition.

## Distribution hypothesis

The expected loop is:

```text
useful diagnostic artifact
-> owner shares evidence with a peer or teammate
-> next qualified diagnostic
-> implementation and expansion
```

This follows the useful parts of Cursor, Harvey, Glean, and Gamma: one daily or
expensive workflow, visible proof, domain trust, and an output that reaches the
next user. It does not copy their funding, breadth, or enterprise sales cost.

Evidence basis:

- canonical private operating note `AI Product Growth Playbook 2026-07-12`;
- public founder and company retrospectives linked from that note.

## Experiments before promotion

1. Interview ten technical owners about repeated context/recovery loss.
2. Require at least three to describe the same expensive job in their own words.
3. Run one bounded diagnostic with a written baseline and acceptance metric.
4. Produce one sanitized before/after report.
5. Test whether that report generates one qualified introduction.

Kill or revise the wedge if ten relevant conversations cannot repeat the pain,
or if the diagnostic cannot produce a measurable result that a client is
willing to reference.

## Metrics

```text
qualified_problem_conversations
repeated_pain_count
diagnostics_started
diagnostics_paid
acceptance_pass_rate
time_to_first_value
pilots_sold
referrals
gross_margin
```

No raw prompts, responses, private logs, credentials, customer messages, or
machine-specific secrets belong in acquisition content or tracked evidence.
